Thanks — this is a strong start, but I want to correct the direction before implementation begins.

Your Persona Intelligence plan is good, but it is still too narrow for what Naya actually needs to become.

The key correction

Naya is not just a persona-aware marketing assistant for one business.

Naya must become a project-aware operating system for entrepreneurs who are often building multiple things at the same time.

That means the current Persona Intelligence work should not be the first architectural layer added in isolation.

It must sit on top of a more important foundation:

Naya must become multi-project and goal-aware first

Right now, from the current architecture, the product is still fundamentally built around:
	•	one user
	•	one Brand DNA
	•	one business context
	•	one global goal logic

That is too limited.

In real use, a Naya user may have:
	•	a main revenue-generating business
	•	a second business or client project
	•	a personal brand
	•	a passion project
	•	an internal project
	•	a personal development or lifestyle project

And these projects do not all behave the same way.

Example:
	•	Project A = agency → must generate money
	•	Project B = content brand → long-term strategic visibility
	•	Project C = passion project → no monetization goal for now

Naya must understand this distinction clearly.

A passion project should not receive the same AI recommendations, urgency, task structure, or performance logic as a revenue-critical business project.

So before you implement the Persona Intelligence System exactly as proposed, I want to revise the architecture.

projects

Each project must have its own:
	•	name
	•	type
	•	description
	•	status
	•	strategic intent
	•	monetization intent
	•	time horizon
	•	priority level
	•	business context

Project types should support examples like:
	•	Business

Each project must also store:
	•	whether it is intended to generate revenue now

	•	whether it is active, paused, or incubating

This changes everything about how AI should reason.

2. Brand DNA cannot remain global-only

The current brand_dna model is useful, but it is currently tied too strongly to the user as a single global identity.

That is not enough.

We need to move toward one of these two models:

Preferred approach
	•	Keep a lightweight global user profile
	•	Add project-level strategy profiles for each project

That means:
	•	one user can have a general operating profile
	•	each project can have its own context and strategic DNA

Because the way I communicate for one project may be completely different from another.

3. Tasks must become project-based

Right now tasks are tied to the user.

That is not sufficient.

We need tasks to belong to a specific project.

Every task should include:
	•	projectId
	•	goal relation
	•	AI source / manual source
	•	task intent
	•	recommended duration
	•	actual duration
	•	scheduling flexibility
	•	energy type if possible
	•	whether it is strategic, executional, admin, content, outreach, or personal

This is important because Naya must be able to answer:
	•	which project is this task serving?
	•	why is this task here?
	•	is it urgent because of revenue or only because of consistency?
	•	is this task aligned with the user’s current objective for this project?

⸻

4. Goals must exist per project

This is another missing core layer.

We need something like:

project_goals

Each project should be able to have:
	•	monthly goals
	•	quarterly goals
	•	milestone goals
	•	non-monetary goals
	•	revenue goals
	•	visibility goals
	•	discipline / consistency goals

Example:
	•	Agency → sign 2 clients this quarter
	•	Content project → publish 3 pieces per week
	•	Passion project → explore direction and create without pressure
	•	Personal project → build consistency, not money

This is extremely important.

Naya must generate tasks and recommendations based on the goal of that specific project, not only on a generic business prompt.
