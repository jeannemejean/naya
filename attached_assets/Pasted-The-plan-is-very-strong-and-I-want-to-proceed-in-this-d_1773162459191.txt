The plan is very strong and I want to proceed in this direction.

It aligns well with what Naya needs to become: not just a task generator, but a real task intelligence system that understands usefulness, sequencing, planning horizons, and user feedback.

Before implementation, I want to add a few important refinements.

1. Treat task feedback as a broader response system, not only deletion

The current task_feedback design is good, but I want it to be future-ready.

Please consider adding a feedbackType field so the system can later support not only deleted tasks, but also:
	•	deleted
	•	dismissed
	•	deferred
	•	reframed

Even if only deleted is used in the first iteration, the table should be designed more broadly than pure deletion.

2. Dependencies should also be generated by AI, not only stored manually

The task_dependencies structure is correct, but I also want dependency generation to be part of task generation.

If the AI generates tasks that clearly follow a sequence, the backend should also be able to create dependency links between them.

Example:
	•	Record video
	•	Edit video
	•	Publish video

This should produce:
	•	Edit video → blocked_by → Record video
	•	Publish video → blocked_by → Edit video

Please include this in the design of generate-daily.

3. Clarify the difference between due date and planning date

For the new planning views (day/week/month), I want you to be careful not to rely only on dueDate.

In Naya, some tasks are true deadlines, while others are simply planned for a given day.

Please keep this distinction in mind in the planning architecture so the calendar does not become semantically confusing.

4. Replan logic must account for blocked tasks

In /api/tasks/replan, I want the logic to explicitly account for dependency chains.

If a prerequisite task is incomplete, downstream tasks should be recognized as blocked and replanned accordingly.

Example:
If “Record video” is still incomplete, then “Edit video” and “Publish video” should not simply remain as normal next steps.

Other than these refinements, the plan is very solid and I want you to proceed.

The direction is correct:
Naya should become a system that not only generates tasks, but understands:
	•	what is useful
	•	what is not useful
	•	what depends on what
	•	what should happen now, later this week, or later this month
	•	how to reorganize work intelligently
