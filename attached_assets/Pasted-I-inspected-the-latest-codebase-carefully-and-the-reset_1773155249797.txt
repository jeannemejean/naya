2. persona_strategy_mapping is not cleaned up

The schema includes a persona_strategy_mapping table, and it references target_personas.id.

But resetUserOnboardingState() does not delete personaStrategyMapping before deleting targetPersonas.

If rows exist in that table, then deleting targetPersonas can fail with a foreign key constraint, which would explain why /api/me/onboarding-reset still returns 500.

Please verify this specifically and add cleanup for persona_strategy_mapping before deleting target_personas.

3. The reset should be wrapped in a database transaction

Right now the reset is a sequence of deletes/updates, but not a transaction.

That means:
	•	if one step fails in the middle,
	•	the workspace can be left in a half-reset state,
	•	and debugging becomes much harder.

Please move resetUserOnboardingState(userId) into a DB transaction so reset becomes atomic.

If one step fails, nothing should be partially deleted.
