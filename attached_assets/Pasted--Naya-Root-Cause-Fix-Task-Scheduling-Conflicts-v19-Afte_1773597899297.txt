# Naya — Root Cause Fix: Task Scheduling Conflicts (v19)

After deep analysis of the v19 codebase, here are the **two root causes** of why tasks overlap in the planning view. Previous fixes only treated symptoms. This document identifies the actual problems and gives exact fixes.

---

## ROOT CAUSE 1 — `generate-daily` overflow creates collisions (PRIMARY)

**File:** `server/routes.ts` — `POST /api/tasks/generate-daily`, the `byDate` loop (~line 3076)

### What is actually happening

The `byDate` loop processes each date in the outer loop. For each date, `curSlot` starts at `latestExistingEndMin` — the end time of the last task already in `existingWeekTasks` for that date.

`existingWeekTasks` is fetched **once, before generation starts** (line 2621). It contains only tasks that were already in the database before this generation call.

When processing Monday's tasks, some tasks may overflow to Tuesday via `genTryOverflowSlot()`. These overflow tasks are **inserted into the DB** (line 3166: `storage.createTask()`). They get real scheduled times on Tuesday — say 09:00 and 09:30.

Then when the outer loop reaches **Tuesday**, it recalculates `existingOnDate` from `existingWeekTasks`. But `existingWeekTasks` was captured before generation — it does NOT include Monday's overflow tasks. So `latestExistingEndMin` for Tuesday = start of day (09:00). `curSlot` starts at 09:00.

Result: Tuesday's own tasks are also placed starting at 09:00 — **directly colliding with Monday's overflow tasks already at 09:00 and 09:30**.

The `assignLanes` function in the frontend splits them into side-by-side columns to avoid visual crash, but the data is wrong: tasks are simultaneously scheduled. The user cannot actually do two tasks at 9:00 AM.

### The fix

Replace the per-date `curSlot` initialization with a **single global `dayNextSlot` Map** that persists across the entire `byDate` loop, including overflow placements.

**Step 1:** Before the `byDate` loop, build `dayNextSlot` from `existingWeekTasks`:

```typescript
// Build global slot tracker seeded from pre-existing tasks
const dayNextSlot = new Map<string, number>();

for (const date of Array.from(byDate.keys())) {
  const genWindow = genGetEffectiveWindow(date);
  const existingOnDate = (existingWeekTasks as any[]).filter(
    (t: any) => t.scheduledDate === date && !t.completed &&
      t.scheduledTime && /^\d{2}:\d{2}$/.test(t.scheduledTime)
  );
  const latestExistingEndMin = existingOnDate.reduce((max: number, t: any) => {
    const endMin = (t.scheduledEndTime && /^\d{2}:\d{2}$/.test(t.scheduledEndTime))
      ? hhmmToMinutes(t.scheduledEndTime)
      : hhmmToMinutes(t.scheduledTime) + (t.estimatedDuration || 30);
    return Math.max(max, endMin);
  }, genWindow.start);

  const isToday = date === todayStr;
  dayNextSlot.set(date, isToday
    ? Math.max(latestExistingEndMin, nowFloorMin + 15)
    : latestExistingEndMin
  );
}
```

**Step 2:** In the inner `for (const pending of dateTasks)` loop, replace per-date `curSlot` usage with `dayNextSlot`:

- Instead of `let curSlot = isToday ? ... : latestExistingEndMin` — remove this variable entirely
- Instead of `curSlot = result.slot + duration` — use `dayNextSlot.set(date, result.slot + duration)`
- When a task overflows to another date (via `genTryOverflowSlot`), update `dayNextSlot` for that overflow date: `dayNextSlot.set(overflowDate, overflowSlot)`
- When reading the current slot for a date: `dayNextSlot.get(date) ?? genWindow.start`

This way, when Tuesday overflows to Wednesday, `dayNextSlot` for Wednesday is updated. When Wednesday's outer loop runs, it starts after the overflow tasks already placed there — **no collision**.

## ROOT CAUSE 2 — `generate-daily` creates duplicate tasks on repeated calls (SECONDARY)

**File:** `server/routes.ts` — `POST /api/tasks/generate-daily`

### What is actually happening

Every call to `generate-daily` calls `storage.createTask()` for every generated task. It never checks whether non-completed tasks already exist for the same project and week.

If the user clicks "Generate my plan" on Monday, then changes a project setting and clicks it again on Wednesday — they now have tasks from run 1 still in the DB (created Monday, never completed), plus tasks from run 2. Both runs assigned times starting from the same baseline. Result: double the tasks, many at the same times.

Looking at the screenshot: Monday shows 7 tasks, several overlapping. This is consistent with a second generation having created new tasks on top of existing ones.

### The fix

At the start of `generate-daily`, before any AI calls, add a **deduplication check**:

```typescript
// Before generating: cancel stale generated tasks for this week to prevent duplicates
// Only cancel tasks with source='generated' that are not completed
// Keep tasks with source='manual' (user created) and source='campaign' (from campaigns)

const weekStartForClean = weekStart; // already computed above
const weekEndForClean = weekEndStr;  // already computed above

const existingGeneratedTasks = (existingWeekTasks as any[]).filter(
  (t: any) => !t.completed && (t.source === 'generated' || t.source === 'ai')
);

// Only clean if there are already generated tasks AND user is requesting a fresh plan
// Check the request body for a flag: if body contains { replaceExisting: true }, clean
const shouldReplace = req.body.replaceExisting === true;

if (shouldReplace && existingGeneratedTasks.length > 0) {
  const projectFilter = projectIdFromBody ? existingGeneratedTasks.filter((t: any) => t.projectId === projectIdFromBody) : existingGeneratedTasks;
  for (const t of projectFilter) {
    await storage.updateTask(t.id, { scheduledDate: null, scheduledTime: null, scheduledEndTime: null } as any);
  }
}
```

**On the frontend**, in the "Generate my plan" button handler (in `planning.tsx`):

If the week already has existing generated (non-completed) tasks for the active project, show a dialog:

```
"You already have [N] tasks scheduled this week for [Project Name].
Replace them with a fresh plan, or add to what's already there?"

[Replace]   [Add to existing]   [Cancel]
```

- If user clicks **Replace**: send `{ replaceExisting: true }` in the request body
- If user clicks **Add to existing**: send `{ replaceExisting: false }` (or omit the field)
- If user clicks **Cancel**: do nothing

This gives the user control while preventing accidental accumulation of duplicate tasks.

## ROOT CAUSE 3 — Overflow date cursor is reset per-day in the outer loop (TERTIARY — fix this too)

In the `byDate` outer loop, `overflowDate` and `overflowSlot` are declared INSIDE the loop:

```typescript
let overflowDate: string | null = null;  // reset each iteration!
let overflowSlot = dayStartMin;          // reset each iteration!
```

This means each date processes its overflow independently. If Monday overflows to Tuesday placing a task at 09:30, and then Wednesday overflows to Tuesday (later in the loop), Wednesday's overflow also starts at `dayStartMin` (09:00) — placing a task at 09:00, before Monday's overflow task at 09:30. Another collision.

**The fix:** Use `dayNextSlot` (from Root Cause 1 fix) for ALL overflow placement. When `genTryOverflowSlot` needs a slot on date D, read from `dayNextSlot.get(D)` and update `dayNextSlot.set(D, newEnd)`. Remove the per-date `overflowDate` / `overflowSlot` / `overflowEnd` / `overflowBreaks` variables and replace with a helper function that uses `dayNextSlot`.

## Summary: What to change

### Change 1 — In `generate-daily` `byDate` loop (Root Cause 1 + 3)

Before the `byDate` outer loop:
- Add global `dayNextSlot` Map seeded from `existingWeekTasks`

Inside the `byDate` outer loop:
- Remove `let curSlot`, `let overflowDate`, `let overflowSlot`, `let overflowEnd`, `let overflowBreaks`, `let dayFull`
- Replace all reads/writes of `curSlot` with `dayNextSlot.get(date)`
- Replace all overflow tracking with reads/writes of `dayNextSlot.get(overflowTargetDate)`
- The `genTryOverflowSlot` function should walk forward through work days, checking `dayNextSlot` for each candidate day

### Change 2 — In `generate-daily` before AI calls (Root Cause 2)

- Add deduplication logic: if `req.body.replaceExisting === true`, null out `scheduledDate/Time/EndTime` on existing generated non-completed tasks for the project/week before generating

### Change 3 — In `planning.tsx` "Generate my plan" button

- Before calling generate-daily: check if non-completed generated tasks already exist this week for the active project
- If yes: show a dialog asking "Replace or Add to existing?"
- Send `replaceExisting: true/false` accordingly

---

## What NOT to change

- Do NOT change `rebalance-week` — that was fixed correctly in v19
- Do NOT change `rebalanceTasksForward` function
- Do NOT change `generate-weekly` (it uses a different mechanism — reschedule + add, not full replace)
- Do NOT change the frontend `assignLanes` function — it is correct as a display fallback, but the data should never require it for newly generated tasks

---

## Testing after the fix

After implementing these changes, test this exact scenario:

1. Open Naya, go to Planning, week view
2. Click "Generate my plan" → tasks appear, all sequential, no overlaps
3. Click "Generate my plan" again → dialog appears asking Replace/Add
4. Click "Replace" → old tasks cleared, new tasks appear, all sequential, no overlaps
5. Click "Rebalance week" once → tasks spread correctly across the week
6. Check no two tasks share the same time slot on the same day

If any two tasks still share the same time slot after step 4, the `dayNextSlot` global tracker is not working correctly.
