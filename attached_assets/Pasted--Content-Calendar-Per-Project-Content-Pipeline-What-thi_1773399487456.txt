 Content Calendar: Per-Project Content Pipeline
What this step does
The Content Calendar currently ignores which project the user is working on — all content from all projects is mixed together. This step makes it fully project-aware, adds a Kanban-style pipeline view (Idea → Draft → Ready → Published), connects AI content generation to the selected project's Brand DNA instead of the global record, and adds a "Generate from strategy" shortcut that pre-fills content briefs from the current week's strategy recommendations.

Task 1 — Server: make /api/content routes project-aware
GET /api/content — update to accept an optional ?projectId= query parameter. When provided, filter results to that project only. When absent, return all content for the user (backward compatible).
In server/storage.ts, update getContent(userId, limit?, projectId?) — add the optional projectId parameter and filter when present.
POST /api/content — already accepts projectId in body. Ensure it is persisted correctly (it is in schema, just verify it flows through).
POST /api/content/generate — currently calls storage.getBrandDna(userId) (global). Fix: when projectId is present in req.body, call storage.getBrandDnaForProject(userId, projectId) instead. Fall back to getBrandDna(userId) only when no projectId is provided.

Task 2 — Schema: add contentStatus field (migration-safe)
The existing status field uses values draft / scheduled / published. We need a richer editorial status. Add a new column to the content table in shared/schema.ts:
tscontentStatus: text("content_status").default("idea"), // idea | draft | ready | published
This is separate from the existing status field (which handles publishing mechanics). contentStatus tracks the editorial workflow. Run npm run db:push.
